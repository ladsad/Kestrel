# Benchmarks

This doc exists so every performance claim on the resume or README traces back to a reproducible number, run on stated hardware, with the exact command used to produce it. No adjectives without a table.

## Methodology

- Load generated by a custom Go harness (`cmd/kestrel-bench`), not a third-party tool, so the read/write mix, key distribution, and payload size are fully controlled and documented per run.
- Every result records: hardware/VM spec, Go version, Kestrel commit hash, command line used, and raw output — committed under `benchmarks/results/<date>-<phase>.md`.
- Latency reported as p50 / p95 / p99, not just average — averages hide tail behavior, which is exactly what distributed systems interviews probe on.

## Planned Benchmark Matrix

| Phase | Benchmark | Metric |
|---|---|---|
| 1 | Single-node throughput, 100% writes, varying concurrency (1/10/50/100 connections) | ops/sec, p50/p95/p99 latency |
| 1 | Single-node throughput, 80/20 read/write mix | ops/sec, p50/p95/p99 latency |
| 2 | AOF replay time vs. log size (10K / 100K / 1M entries) | seconds to recover |
| 2 | Write latency by fsync policy (`always` vs `everysec` vs `no`) | p50/p95/p99 latency delta |
| 3 | 3-node cluster throughput vs. single node | ops/sec delta, replication lag under load |
| 4 | Time-to-new-leader after simulated leader kill | seconds, over 10 trials |
| 4 | Time-to-writes-resumed after leader kill | seconds, over 10 trials |
| 4 | Cluster throughput during an election (degradation window) | ops/sec over time, graphed |
